#+title: Runbook: self-hosted binary cache (cache.rytswd.com)
#+filetags: :runbook:infra:

This flake substitutes from, and pushes to, a self-hosted binary cache. Reads and writes are structurally different paths that fail independently, and that asymmetry drives everything below: the read side is already live and costs nothing to depend on, while the write side is private infrastructure that can lag behind without hurting anyone. Design rationale lives in [[file:../../air/v0.1/binary-cache-niks3.org]].

* The shape

| Concern                 | Value                                                                                                              |
|-------------------------+--------------------------------------------------------------------------------------------------------------------|
| Substituter (read path) | =https://cache.rytswd.com= -- Cloudflare R2 behind a custom domain; no server on this path                        |
| Write path              | niks3 server (presigned uploads, reference tracking, GC) -- private infrastructure, endpoint via =NIKS3_SERVER_URL= |
| Signing key             | =cache.rytswd.com-1:/1R/t3GTiG7HVo2Xpl+f0CBhrZNBwFMQSYfaxd3sSSc=                                                   |

Reads never touch the niks3 server: a write-path outage (or the write-path host simply not existing yet) only pauses uploads, never substitution. This is why the substituter could be wired into every consumer before the server endpoint was even decided.

* Geography: one bucket + CDN

There is exactly one bucket, and that is a decision rather than an omission. The custom domain places R2 behind Cloudflare's global CDN, so every read is served from the edge nearest the consumer, and R2-to-edge egress is free -- a cached read costs nothing no matter where it lands. Per-region buckets would mean self-managed replication and geo-routing to solve a problem the CDN already solves. The ecosystem precedent is the same shape: =cache.nixos.org= is a single S3 region behind Fastly, and this flake already consumes that pattern from the read side -- =hetzner-cache.numtide.com= is a mirror serving paths signed with the =cache.nixos.org-1= key.

The edge-cache policy follows directly from how the two object kinds behave:

| Path        | Access pattern                                      | Edge policy                                  |
|-------------+-----------------------------------------------------+----------------------------------------------|
| =nar/*=     | content-addressed, immutable                        | long TTL -- cache as aggressively as possible |
| =*.narinfo= | immutable once written, but probed for every lookup | cacheable; 404s short (<=1 min) or uncached  |

The 404 line is the one that matters operationally: nix probes substituters for paths they mostly don't have, so negative lookups are constant -- and a freshly pushed path must become visible promptly. A long-cached 404 would make a new push invisible until the negative entry expired, which is why misses get a short TTL (or none) while hits on immutable objects get a long one.

** One-time Cloudflare setup

- Bucket location hint near the writer (the provisioner today, the niks3 write-path host when it lands): writes and cold-read fills are the only traffic that reaches the bucket's home region; everything else is edge-served, so the hint optimises for the writer, not the readers.
- A "cache everything" Cache Rule on the domain: NAR and narinfo paths don't match Cloudflare's default cacheable-extension list, so without this rule nothing is edge-cached at all.
- Edge TTLs per the table above: long for =nar/*=, modest for =*.narinfo=, short or none for 404s.
- Smart Tiered Cache on: edge misses fill through the tier nearest the bucket instead of every edge hitting origin independently.

** Regional escape hatch: mirror, never a second bucket

If a region's latency or hit rate ever genuinely hurts, the answer is a pull-through mirror, not another bucket: a box in that region fronting =https://cache.rytswd.com= (the nginx =proxy_cache= shape, the same thing the Numtide hetzner mirror is to =cache.nixos.org=), listed first in that region's substituters. Signatures travel inside the narinfos, so the same =cache.rytswd.com-1= key verifies regardless of which host serves the bytes -- a mirror needs no key material and no trust changes anywhere. A second bucket, by contrast, reintroduces exactly the replication and routing burden the single-bucket decision avoided. The same policy carries over per-tenant to the future multi-tenant cache service: one bucket per tenant, regional needs answered by mirrors.

* Reading from it

Nothing to do. The substituter and public key are wired in the two places that matter: =flake.nix= =nixConfig= (the only configuration a fresh machine running =nix run github:rytswd/nix-config= ever sees) and =nixos-config/modules/nix-base.nix= (the system-level layered cache model for machines built from this flake). Until the first push lands, the bucket serves no =nix-cache-info= and nix logs a "does not appear to be a binary cache" warning -- harmless, it falls through to the other caches.

* Pushing

Provisioner-class machines push -- the ones that already hold private-repo access and build for others (see "Who needs what" in [[file:remote-provision.org]]). They are the only holders of the niks3 API token; thin machines and coder-class workspaces never see it.

#+begin_src bash
export NIKS3_SERVER_URL=https://<niks3-server>   # write-path endpoint; see "Custody" below
export NIKS3_AUTH_TOKEN_FILE=/path/to/token      # or place the token at $XDG_CONFIG_HOME/niks3/auth-token
nix run .#cache-push -- <store-path|installable...>
#+end_src

The app resolves its arguments to store paths (=./result= links and flake installables both work), always prints the fully-resolved =niks3 push= command before executing -- same contract as =provision= / =deploy= -- and =--dry-run= stops right there. Both variables are the niks3 CLI's own; the app only adds early, pointed failures when they are missing. Any other flag is passed through to =niks3 push= verbatim.

The write-path endpoint is deliberately not recorded in this public repo: the host is still being decided (Air doc History), and it stays private infrastructure once it is. The value belongs with the other provisioner-class material in the private repo.

* Push after build

Make pushing the closing move of every build that another machine might want: after a =provision= or =deploy= ([[file:remote-provision.org]]), push the system closure you just built; the same goes for HM profiles and darwin builds -- the cache is multi-arch by nature and the MBPs push their own closures.

#+begin_src bash
nix build .#nixosConfigurations.<host>.config.system.build.toplevel
nix run .#cache-push -- ./result
#+end_src

The provisioner builds once, pushes once, and every subsequent machine -- including the target itself on a later self-rebuild -- substitutes instead of rebuilding. That habit is what keeps thin machines thin.

* Custody

The =cache.rytswd.com-1= secret key lives in the private repo as a provisioner-class sops secret with YubiKey-only recipients, and is loaded by the niks3 /server/ (=services.niks3.signKeyFiles= upstream) -- narinfos are signed server-side at upload. Neither =cache-push= nor any pushing machine ever holds signing key material: a pusher carries only the API token, so a leaked token is revoked by a server-side config change while the signing key never leaves the write-path host.

The API token is provisioner-class in the same way: YubiKey-only sops in the private repo, handed only to machines that build for others.

* Cachix retirement

The =rytswd-nix-config.cachix.org= substituter and key stay in =flake.nix= =nixConfig= (plus the matching allow-list entries in =nix-base.nix=) until this cache holds the hot set -- in practice a few days of normal pushes covering the standard profiles: system closures of the active hosts and the coder HM profile. When that holds, remove the two =nixConfig= entries and the =nix-base.nix= pair, and rewrite this section to record the retirement. The community caches (ghostty, niri, ...) are unaffected -- they serve closures this cache never will.
