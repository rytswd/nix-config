# My Style

## Commit messages
- Unless the repo already uses Conventional Commits, I want to keep the commit message simple:
  - Examples: "Support new subcommand airctl context"
  - When the project is quite involved and already using conventional commit, use format like:
    - `feat(air-core): improve traversal performance`
- Subject ≤ 72 chars, imperative mood, no trailing period.
- Body: wrap at 72 chars; explain *why* before *what*. Skip the body
  for trivial changes.
- No "Generated by", "Co-authored-by Claude", or similar agent
  attribution lines.

## PR descriptions
- Three implicit sections, in order:
  1. **Why** — the problem this solves.
  2. **What** — the high-level shape of the change.
  3. **How to verify** — concrete commands or steps a reviewer can run.
- Link to issues / Air docs by relative path when applicable.
- No screenshots-by-default; add only when the change is visual.

## Documentation
- Org-mode for personal / repo-internal docs (Air documents,
  README.org).
- Markdown only when required by tooling (GitHub-rendered files
  that aren't .org-aware, NPM/PyPI READMEs, agent skill `SKILL.md`).
- Tables, lists, and short paragraphs over long prose.
- Cross-link related docs by relative path; avoid duplicating content.

## Agent response style
- **Terse by default.** No preamble ("I'll help you with that…"),
  no recap of what the user just said.
- Show diffs, file paths, and concrete commands; avoid narrating what
  the diff already shows.
- Lists / tables for enumerable content; prose only when explaining
  reasoning or trade-offs.
- When proposing options, name them clearly and give a one-line
  recommendation rather than presenting equal alternatives.
- Surface tradeoffs and uncertainty explicitly — don't hedge with
  "perhaps" or "maybe" when a definitive statement is possible.
