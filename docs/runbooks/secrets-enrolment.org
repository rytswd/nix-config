#+title: Runbook: ephemeral-instance secrets enrolment
#+filetags: :runbook:secrets:

* Policy
YubiKey-backed age is the primary and default decryption mechanism everywhere. The core secret set — the private repo's existing primary sops files, the ones every YubiKey machine already decrypts — is encrypted to YubiKey-derived and per-host recipients only, always. What this runbook adds is a deliberately narrow second tier: hosts that are trusted enough to hold /some/ secrets but physically cannot present a YubiKey (remote workspaces, cloud instances) get a per-instance age recipient that is enrolled into, and revoked from, the private repo's /ephemeral class/ (=ephemeral/**=) only. Enrolment is scoped, tracked, and reversible — the lifecycle is the mechanism, not the key trick. Design rationale lives in [[file:../../air/v0.1/secrets-host-enrolment.org]].

If a host does not clearly need this tier, it does not get it: long-lived physical machines use a YubiKey, fully-untrusted machines (coder-class workspaces) keep =local.secrets.enable = false= and receive nothing.

* Where the tooling lives
The =enroll= / =revoke= apps live in the *private repo*, run from its checkout:

#+begin_src sh
cd <ghRoot>/rytswd/nix-config-private
nix run .#enroll -- <args>   # see its --help and the private repo's docs
nix run .#revoke -- <name>
#+end_src

They used to live here as =secrets-enroll= / =secrets-revoke=, but the tooling edits the private repo's =.sops.yaml= and registry, so every layout change there had to be mirrored in a public repo that cannot see the files it edits — that coupling produced real drift (this runbook's earlier revisions mis-stated the standing recipient set and described a =core/**= path that never existed). Mechanics — sentinel block format, registry shape, dry-run behaviour, what gets committed — are documented next to the tooling in the private repo; this runbook keeps only the lifecycle and the public-side wiring.

Both apps mutate a checkout you already trust, print their changes, and commit locally — they never push. Enrolment must run from a machine that can already decrypt the ephemeral class (in practice a YubiKey machine): =sops updatekeys= must unwrap each file's data key before re-wrapping it for the new recipient set, so it cannot run from a keyless host.

* Secret classes
Two classes in the private repo's =.sops.yaml=, and this is the whole trust model:

- *Core class* :: The repo's existing primary sops files. Recipients are the standing set only; never touched by the enrolment tooling, never gains per-instance recipients.
- *Ephemeral class* (=ephemeral/**=) :: The standing recipients /plus/ the currently-enrolled instances. Holds only what keyless instances genuinely need (e.g. the atuin sync endpoint) — never the core set.

The /standing/ recipients of the ephemeral class mirror the full recipient set the core sops-nix files use: the pinless YubiKey set (sops-nix cannot prompt for a PIN at activation), the legacy slot key, and the per-host machine keys. An earlier revision of this runbook described an admin-YubiKey-only standing list — wrong, and exactly the kind of drift that motivated moving the tooling next to the file it edits. What enrolment adds on top is only the per-instance recipients, maintained inside a sentinel-comment block by the apps.

* Lifecycle
Enrol immediately after provisioning, use for the instance's lifetime, revoke as part of teardown. Every enrolment without a matching revocation is drift, and the registry exists to make that drift visible at a glance.

** Enrol — NixOS-class instance
The instance's own ed25519 SSH host key doubles as its age identity (=ssh-keyscan | ssh-to-age=), so nothing is generated on or copied to the target. Note =ssh-to-age= understands ed25519 keys only — NixOS generates one by default, so this only bites unusual sshd configs.

Run =nix run .#enroll= from the private-repo checkout, pointing it at the instance. On the instance side, decryption happens at activation via the host key (=sops.age.sshKeyPaths= — wired in the private repo's host config).

** Enrol — HM-standalone-class workspace
For workspaces with no system-level access (no host key to scan, no =/etc/ssh= to read at activation), generate a user-level age identity on the target instead:

#+begin_src sh
age-keygen -o ~/.config/sops/age/keys.txt   # on the instance
#+end_src

That path is the Home Manager sops-nix default. The identity never leaves the instance; only the public recipient travels back, passed to =nix run .#enroll= in the private-repo checkout.

** Public-side wiring (both classes)
After enrolment, set the tier in the host's profile in this repo:

#+begin_src nix
local.secrets.tier = "ephemeral";
#+end_src

Ephemeral-class modules (e.g. atuin sync) now activate; core-class modules fall back exactly as if secrets were disabled. See =user-config/modules/lib/secrets.nix= for the gating contract. If the host's profile imports the private repo's sops-nix HM module, check where that module points =sops.age.keyFile= — it targets the YubiKey-machine layout, so an HM-standalone-class instance whose identity lives at the =age-keygen= default above needs an explicit override (see =user-config/ryota/server.nix= for the worked example).

** Use
Nothing ongoing. The instance decrypts ephemeral-class secrets at every activation; YubiKey machines continue to decrypt everything. If the instance starts needing a secret that lives in the core class, that is a design conversation, not an enrolment — the core class never gains per-instance recipients.

** Revoke — instance teardown
Revocation is a mandatory step in every teardown runbook for an enrolled instance (cloud instance termination, workspace deletion): =nix run .#revoke -- <name>= from the private-repo checkout. It drops the recipient, fills the =Revoked= column of the registry row (rows are never deleted — the registry plus its git history is the audit log), re-runs =sops updatekeys=, and commits. Review and push.

Know what =updatekeys= does /not/ do: it re-wraps data keys for the new recipient set but does not rotate them, so an instance that kept a copy of the old encrypted files can decrypt those copies forever. For a friendly teardown (instance simply retired) that is acceptable. If the instance may have been compromised, follow up with =sops rotate -i= on each ephemeral-class file /and/ rotate the underlying secret values themselves.

* Recreated instances — the fork
Cloud instances and workspaces get a brand-new SSH host key every time they are recreated, which invalidates their enrolled recipient. Two paths, decided at provision time:

- *Seed the old host key* :: Provision with the remote-provisioning flow's =--seed-host-key= (see [[file:../../air/v0.1/remote-provisioning.org]]), injecting the previously-enrolled ed25519 host key via =--extra-files=. The instance keeps its age identity: no =.sops.yaml= change, no =updatekeys=, no private-repo commit. The cost is that private key material transits the provisioning machine — acceptable when the provisioner is a trusted personal machine and the window is short, but it is a real handling of a real private key.
- *Fresh key, revoke + enrol* :: Let the instance generate a new host key, then revoke followed by enrol. Cleaner — no key material ever leaves the instance — at the cost of one =updatekeys= round and a registry row of churn. The registry deliberately keeps both rows, so recreation frequency stays visible.

Default to the fresh-key path; reach for seeding when re-provisioning is frequent enough that the sops churn becomes noise, and record that choice in the instance's own notes.

* Periodic sweep — registry vs reality
Stale recipients are both hygiene and security debt, so reconcile the registry against the actual fleet on a regular cadence (and opportunistically whenever an enrolment or revocation has you in the file anyway):

1. List active rows (empty =Revoked= column) in =ephemeral/registry.org= in the private repo.
2. For each, confirm the instance still exists and still needs its secrets.
3. Anything that fails the check: =nix run .#revoke -- <name>= from the private-repo checkout.

Also eyeball the size of the enrolled block in =.sops.yaml= while there. It should stay small enough to read in one glance — if it grows past a handful of entries, that is the signal the fleet needs a different trust model, not more enrolments.
