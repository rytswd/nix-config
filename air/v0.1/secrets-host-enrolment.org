#+title: Secrets: YubiKey-primary with scoped per-instance enrolment
#+state: work-in-progress
#+FILETAGS: :secrets:

* Summary
Keep YubiKey-backed age as the primary and default decryption mechanism everywhere, and add a narrowly-scoped second tier for hosts that can never see a YubiKey (remote workspaces, cloud instances): per-instance age recipients derived from the instance's own SSH key, enrolled and revoked through a scripted, tracked runbook. The recipient registry and re-encryption live in the private repo; this repo carries only the generic tooling and the runbook.

* Motivation
Today secret handling is binary: a YubiKey-derived age key decrypts everything, or =local.secrets.enable = false= and the host gets nothing (the coder fallback). That leaves no answer for hosts that are trusted enough to hold /some/ secrets but physically cannot present a YubiKey -- remote workspaces and cloud instances like the EC2 devbox.

The sharp edge called out in review: these instances get a /brand-new SSH host key every time they are recreated/. Any per-instance recipient scheme lives or dies by how recreation is tracked -- stale recipients accumulating in =.sops.yaml= is both a hygiene and a security problem. So the enrolment lifecycle (enrol → use → revoke on teardown) is the core deliverable, not the key derivation trick.

Why not a passphrase-protected master identity as break-glass: a portable software copy of the master defeats the point of hardware binding, and the preference is explicit -- YubiKey as much as possible, host-specific only where unavoidable.

** Goals
- YubiKey recipients remain the only recipients for the core secret set; nothing about existing hosts changes.
- A scoped secret class (e.g. =ephemeral/= in the private repo's =.sops.yaml= creation rules) encrypted to YubiKeys /plus/ currently-enrolled instance keys -- containing only what those hosts genuinely need (e.g. atuin sync key, selected service tokens), never the core set.
- =nix run .#secrets-enroll -- <name> <ssh-target>= and =nix run .#secrets-revoke -- <name>=: derive the age recipient (=ssh-keyscan | ssh-to-age= for NixOS hosts; generated user-level age key for HM-standalone-only workspaces), update =.sops.yaml=, run =sops updatekeys= over the scoped files, and record the enrolment in a registry file -- all inside a private-repo checkout, committed as one change.
- A registry (name, recipient, created date, purpose) in the private repo so =.sops.yaml= drift is visible at a glance; revocation is an explicit step in the instance-teardown runbooks (devbox terminate, workspace deletion).
- Clear runbook covering both directions, cross-linked from the provisioning runbook (recreated instance → new host key → re-enrol).

** Non-Goals
- Changing the YubiKey age flow, the GPG/FIDO2 signing fallbacks (=vcs/git/yubikey.nix=), or =shared/keys.nix=.
- Per-host keys for long-lived physical machines -- they have YubiKey access by definition.
- Secrets on fully-untrusted machines; =local.secrets.enable = false= remains the right answer there (coder class).
- The actual =.sops.yaml= contents and secret values -- private repo concern; this doc fixes the mechanism and interface only.

* Proposal
Three layers:

1. *Scoping (private repo)*: split =.sops.yaml= creation rules into =core/**= (YubiKey recipients only) and =ephemeral/**= (YubiKey recipients + the enrolled-instance group). Modules in this repo that reference ephemeral-class secrets point at the =ephemeral/= sops files; everything else keeps pointing at core files and simply stays unavailable on enrolled-but-keyless hosts.
2. *Enrolment tooling (this repo, generic)*: the =secrets-enroll= / =secrets-revoke= apps operate on a private-repo checkout located via =local.repoPath= conventions. For NixOS targets the recipient comes from the host SSH key (=ssh-to-age=), decrypted at activation via =sops.age.sshKeyPaths=. For HM-standalone workspaces with no system access, =age-keygen= on the target, recipient captured, identity stays at =~/.config/sops/age/keys.txt= (the HM sops-nix default).
3. *Lifecycle (runbook)*: enrol immediately after provision (the provision runbook links here); revoke as part of teardown; a periodic "registry vs reality" sweep -- any registry entry whose instance no longer exists gets revoked and =updatekeys= re-run. Recreating an instance is exactly revoke-then-enrol; the registry entry's history shows the churn.

A third =local.secrets= mode ties it together in HM: today the toggle is all-or-nothing; it gains awareness that a host may have a non-YubiKey key serving only ephemeral-class secrets, so core-secret modules fall back gracefully while ephemeral-secret modules activate.

* Design Details
- =ssh-to-age= only works for ed25519 keys -- fine for NixOS defaults; the runbook notes it.
- Ordering on fresh NixOS installs: sops-nix needs the host key at activation; with [[file:remote-provisioning.org]]'s optional =--seed-host-key=, a recreated instance can /reuse/ its enrolled identity instead of forcing re-enrolment -- the runbook presents both paths (seed = no sops churn; fresh key = enrol again) with the trade-off (seeding copies private key material through the provisioner; fresh is cleaner, costs one =updatekeys=).
- The enrolled-instance group in =.sops.yaml= should stay small enough to eyeball -- if it grows past a handful, that is the signal the fleet needs a different trust model, not more enrolments.
- Public-repo constraint: scripts and runbook are fully generic ("remote workspace", "cloud instance"); names, hosts, and the registry live privately.

** Test Plan
- Enrol a throwaway VM, confirm ephemeral-class decryption on it and continued core-class decryption on a YubiKey machine; revoke; confirm the VM can no longer decrypt newly re-encrypted files.
- Verify =sops updatekeys= round-trip leaves YubiKey decryption intact (the failure that would actually hurt).

* Drawbacks
- =updatekeys= churn in the private repo on every enrol/revoke; acceptable at this fleet size and it doubles as an audit log.
- Enrolled instances can decrypt the ephemeral class while compromised; scoping is the containment, revocation the remedy.

* Alternatives
- /Per-host keys for everything/: rejected -- dilutes hardware binding for hosts that do not need it.
- /Passphrase-protected master identity/: rejected as above; revisit only if a true no-network no-YubiKey recovery scenario emerges, and then as a sealed offline artifact, not a daily mechanism.
- /Secrets-over-SSH at runtime (fetch from a YubiKey machine on demand)/: avoids enrolment but makes a laptop an availability dependency for cloud hosts; wrong coupling direction.

* Infrastructure Needed
None beyond a private-repo checkout on the enrolling machine; =ssh-to-age= and =age= already ship in the relevant module sets.
