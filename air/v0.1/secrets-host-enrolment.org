#+title: Secrets: YubiKey-primary with scoped per-instance enrolment
#+state: complete
#+FILETAGS: :secrets:

* Summary
Keep YubiKey-backed age as the primary and default decryption mechanism everywhere, and add a narrowly-scoped second tier for hosts that can never see a YubiKey (remote workspaces, cloud instances): per-instance age recipients derived from the instance's own SSH key, enrolled and revoked through a scripted, tracked runbook. The recipient registry, re-encryption, and (since the 2026-06-29 relocation, see History) the enrol/revoke tooling live in the private repo; this repo carries the lifecycle runbook and the =local.secrets= tier wiring.

* Motivation
Today secret handling is binary: a YubiKey-derived age key decrypts everything, or =local.secrets.enable = false= and the host gets nothing (the coder fallback). That leaves no answer for hosts that are trusted enough to hold /some/ secrets but physically cannot present a YubiKey -- remote workspaces and cloud instances like the EC2 devbox.

The sharp edge called out in review: these instances get a /brand-new SSH host key every time they are recreated/. Any per-instance recipient scheme lives or dies by how recreation is tracked -- stale recipients accumulating in =.sops.yaml= is both a hygiene and a security problem. So the enrolment lifecycle (enrol → use → revoke on teardown) is the core deliverable, not the key derivation trick.

Why not a passphrase-protected master identity as break-glass: a portable software copy of the master defeats the point of hardware binding, and the preference is explicit -- YubiKey as much as possible, host-specific only where unavoidable.

** Goals
- YubiKey recipients remain the only recipients for the core secret set; nothing about existing hosts changes.
- A scoped secret class (e.g. =ephemeral/= in the private repo's =.sops.yaml= creation rules) encrypted to YubiKeys /plus/ currently-enrolled instance keys -- containing only what those hosts genuinely need (e.g. atuin sync key, selected service tokens), never the core set.
- =nix run .#enroll= / =nix run .#revoke= (in the private repo, run from its checkout): derive the age recipient (=ssh-keyscan | ssh-to-age= for NixOS hosts; generated user-level age key for HM-standalone-only workspaces), update =.sops.yaml=, run =sops updatekeys= over the scoped files, and record the enrolment in a registry file -- all inside a private-repo checkout, committed as one change.
- A registry (name, recipient, created date, purpose) in the private repo so =.sops.yaml= drift is visible at a glance; revocation is an explicit step in the instance-teardown runbooks (devbox terminate, workspace deletion).
- Clear runbook covering both directions, cross-linked from the provisioning runbook (recreated instance → new host key → re-enrol).

** Non-Goals
- Changing the YubiKey age flow, the GPG/FIDO2 signing fallbacks (=vcs/git/yubikey.nix=), or =shared/keys.nix=.
- Per-host keys for long-lived physical machines -- they have YubiKey access by definition.
- Secrets on fully-untrusted machines; =local.secrets.enable = false= remains the right answer there (coder class).
- The actual =.sops.yaml= contents and secret values -- private repo concern; this doc fixes the mechanism and interface only.

* Proposal
Three layers:

1. *Scoping (private repo)*: split =.sops.yaml= creation rules into =core/**= (YubiKey recipients only) and =ephemeral/**= (YubiKey recipients + the enrolled-instance group). Modules in this repo that reference ephemeral-class secrets point at the =ephemeral/= sops files; everything else keeps pointing at core files and simply stays unavailable on enrolled-but-keyless hosts.
2. *Enrolment tooling (private repo, next to the files it edits)*: the =enroll= / =revoke= apps operate on the checkout they live in (originally implemented here as flake apps driving a separate private checkout; relocated 2026-06-29 -- see History for the coupling evidence). For NixOS targets the recipient comes from the host SSH key (=ssh-to-age=), decrypted at activation via =sops.age.sshKeyPaths=. For HM-standalone workspaces with no system access, =age-keygen= on the target, recipient captured, identity stays at =~/.config/sops/age/keys.txt= (the HM sops-nix default).
3. *Lifecycle (runbook)*: enrol immediately after provision (the provision runbook links here); revoke as part of teardown; a periodic "registry vs reality" sweep -- any registry entry whose instance no longer exists gets revoked and =updatekeys= re-run. Recreating an instance is exactly revoke-then-enrol; the registry entry's history shows the churn.

A third =local.secrets= mode ties it together in HM: today the toggle is all-or-nothing; it gains awareness that a host may have a non-YubiKey key serving only ephemeral-class secrets, so core-secret modules fall back gracefully while ephemeral-secret modules activate.

* Design Details
- =ssh-to-age= only works for ed25519 keys -- fine for NixOS defaults; the runbook notes it.
- Ordering on fresh NixOS installs: sops-nix needs the host key at activation; with [[file:remote-provisioning.org]]'s optional =--seed-host-key=, a recreated instance can /reuse/ its enrolled identity instead of forcing re-enrolment -- the runbook presents both paths (seed = no sops churn; fresh key = enrol again) with the trade-off (seeding copies private key material through the provisioner; fresh is cleaner, costs one =updatekeys=).
- The enrolled-instance group in =.sops.yaml= should stay small enough to eyeball -- if it grows past a handful, that is the signal the fleet needs a different trust model, not more enrolments.
- Public-repo constraint: scripts and runbook are fully generic ("remote workspace", "cloud instance"); names, hosts, and the registry live privately.
- Prior-art contrast (Mic92/dotfiles): sops-nix's own author commits ciphertext, =.sops.yaml=, and vars publicly -- valid under sops' threat model and operationally simpler (no private checkout to coordinate). Kept private here on purpose: public ciphertext reveals the secret /inventory/ (file names ≈ which services and machines exist), the recipient set and its churn over git history (machine count, enrolment timeline -- exactly what our registry tracks, which is why the registry is private too), and donates an offline-attack corpus should any recipient key ever weaken. The enrolment tooling consequently operates on a private checkout rather than in-repo files; that is a feature, not an inconvenience.
- Related leak surface found during implementation: sops-nix /template/ placeholders are =sha256(secret name)=, so even with private sops files, any public module defining =sops.templates= lets a reader map template → secret name → class from public material alone. Harmless for the current names, but secret /names/ are part of the metadata surface -- name them blandly in public modules, and keep anything whose existence is sensitive defined in the private repo's modules instead.
- Per-module placement rule (fixed at the 2026-06-29 relocation, generalising the previous point): a module may stay public when it consumes only secret /values/ -- template placeholders, secret output paths -- because a bland secret name is already accepted metadata surface; a module moves behind a private seam when the /structure or existence/ of its secrets is itself the sensitive part (which accounts exist, which services are subscribed). The email module is the exemplar of the second kind: the public communication bundle only guards on "at least one account exists" (=user-config/modules/communication/email.nix=) while every account definition lives in the private repo's module. Atuin was assessed against the rule and stays public as value-only: its template embeds =sha256(name)= placeholders from a literal file, so it reveals one bland name and -- because nothing dereferences the secret at eval -- cannot break a degraded (stub-override) eval even when the private module that defines the secret is a no-op; the considered =config.sops.secrets ? "…"= gate proved unnecessary and was not added.

** Test Plan
- Enrol a throwaway VM, confirm ephemeral-class decryption on it and continued core-class decryption on a YubiKey machine; revoke; confirm the VM can no longer decrypt newly re-encrypted files.
- Verify =sops updatekeys= round-trip leaves YubiKey decryption intact (the failure that would actually hurt).

* Drawbacks
- =updatekeys= churn in the private repo on every enrol/revoke; acceptable at this fleet size and it doubles as an audit log.
- Enrolled instances can decrypt the ephemeral class while compromised; scoping is the containment, revocation the remedy.

* Alternatives
- /Per-host keys for everything/: rejected -- dilutes hardware binding for hosts that do not need it.
- /Passphrase-protected master identity/: rejected as above; revisit only if a true no-network no-YubiKey recovery scenario emerges, and then as a sealed offline artifact, not a daily mechanism.
- /Secrets-over-SSH at runtime (fetch from a YubiKey machine on demand)/: avoids enrolment but makes a laptop an availability dependency for cloud hosts; wrong coupling direction.

* Infrastructure Needed
None beyond a private-repo checkout on the enrolling machine; =ssh-to-age= and =age= already ship in the relevant module sets.

Devbox wired to the tier the enrolment created for it: =local.secrets.tier = "ephemeral"= in the server profile, the private sops-nix HM module imported (as =nixos.nix= already does), and =sops.age.keyFile= forced to the HM sops-nix default =~/.config/sops/age/keys.txt= because the private module points it at the YubiKey-machine layout that nothing populates on an enrolled instance -- the user-level key generated at enrolment is the box's only identity. This deliberately reverses the keyless-eval side effect noted in the Design Divergence below: the server profile forces the private input again, accepted because the provisioner that builds it holds private-repo access anyway, while keyless machines keep the =stubs/nix-config-private= override for degraded evals. Validated both ways (real input, and stub override with SSH masked); the coder profile's derivation is byte-identical before/after. Known follow-up: =flake.lock= pins the private input at a rev predating the ephemeral class, so the devbox cannot decrypt the atuin endpoint at activation until the lock is bumped after the private side lands (lock edits are out of scope here).

** Design Divergence
- Non-Goals lists =vcs/git/yubikey.nix= as unchanged, but the implementation gates its =sops.secrets= stub definitions (and =vcs/jj='s config template) behind the new =coreAvailable= flag. The signing/auth flow on YubiKey hosts is untouched; without the gate, however, no host importing the vcs bundle could ever take the ephemeral tier -- the bundle defined core-class secrets unconditionally, which is exactly the "core-secret modules fall back gracefully" requirement in the Proposal. Side effect worth noting: the headless-server HM profile now evaluates without forcing the private flake input.
