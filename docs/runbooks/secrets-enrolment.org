#+title: Runbook: ephemeral-instance secrets enrolment
#+filetags: :runbook:secrets:

* Policy
YubiKey-backed age is the primary and default decryption mechanism everywhere. The core secret set is encrypted to YubiKey recipients only, always. What this runbook adds is a deliberately narrow second tier: hosts that are trusted enough to hold /some/ secrets but physically cannot present a YubiKey (remote workspaces, cloud instances) get a per-instance age recipient that is enrolled into, and revoked from, the private repo's /ephemeral class/ only. Enrolment is scoped, tracked, and reversible — the lifecycle is the mechanism, not the key trick. Design rationale lives in [[file:../../air/v0.1/secrets-host-enrolment.org]].

If a host does not clearly need this tier, it does not get it: long-lived physical machines use a YubiKey, fully-untrusted machines (coder-class workspaces) keep =local.secrets.enable = false= and receive nothing.

* Preconditions
- A machine that can already decrypt the ephemeral class — in practice a YubiKey machine. =sops updatekeys= must unwrap each file's data key before re-wrapping it for the new recipient set, so enrolment cannot run from a keyless host.
- A local checkout of the private repo at =<ghRoot>/rytswd/nix-config-private= (the =local.ghRoot= convention from =user-config/modules/lib/paths.nix=; =--private-repo <path>= to override). The tooling never clones — operating on a checkout you already trust is part of the point.
- The private repo laid out per [[*Private-repo interface][Private-repo interface]] below.
- For NixOS-class targets: the instance reachable over SSH with an ed25519 host key (the NixOS default). For HM-standalone-class targets: a shell on the instance to run =age-keygen=.

* Lifecycle
Enrol immediately after provisioning, use for the instance's lifetime, revoke as part of teardown. Every enrolment without a matching revocation is drift, and the registry exists to make that drift visible at a glance.

** Enrol — NixOS-class instance
The instance's own ed25519 SSH host key doubles as its age identity (=ssh-keyscan | ssh-to-age=), so nothing is generated on or copied to the target.

#+begin_src sh
nix run .#secrets-enroll -- <name> <ssh-target> --purpose "<why this instance needs secrets>"
#+end_src

The app prints every planned mutation as a unified diff before applying anything (=--dry-run= stops at the printing), then commits the private checkout as one change. It never pushes — review the diff and push yourself.

Note =ssh-to-age= understands ed25519 keys only; the app detects an RSA/ECDSA-only target and says so. NixOS generates an ed25519 host key by default, so this only bites unusual sshd configs.

On the instance side, decryption happens at activation via the host key (=sops.age.sshKeyPaths= pointing at =/etc/ssh/ssh_host_ed25519_key= — wired in the private repo's host config). Then set the tier in the host's profile in this repo:

#+begin_src nix
local.secrets.tier = "ephemeral";
#+end_src

Ephemeral-class modules (e.g. atuin sync) now activate; core-class modules fall back exactly as if secrets were disabled. See =user-config/modules/lib/secrets.nix= for the gating contract.

** Enrol — HM-standalone-class workspace
For workspaces with no system-level access (no host key to scan, no =/etc/ssh= to read at activation), generate a user-level age identity on the target instead:

#+begin_src sh
age-keygen -o ~/.config/sops/age/keys.txt   # on the instance
#+end_src

That path is the Home Manager sops-nix default, so no extra wiring is needed. The identity never leaves the instance; only the public recipient travels back:

#+begin_src sh
nix run .#secrets-enroll -- <name> --age-key age1... --purpose "<why>"
#+end_src

** Use
Nothing ongoing. The instance decrypts ephemeral-class secrets at every activation; YubiKey machines continue to decrypt everything. If the instance starts needing a secret that lives in the core class, that is a design conversation, not an enrolment — the core class never gains non-YubiKey recipients.

** Revoke — instance teardown
Revocation is a mandatory step in every teardown runbook for an enrolled instance (cloud instance termination, workspace deletion):

#+begin_src sh
nix run .#secrets-revoke -- <name>
#+end_src

This drops the recipient from =.sops.yaml=, fills the =Revoked= column of the registry row (rows are never deleted — the registry plus its git history is the audit log), re-runs =sops updatekeys=, and commits. Review and push.

Know what =updatekeys= does /not/ do: it re-wraps data keys for the new recipient set but does not rotate them, so an instance that kept a copy of the old encrypted files can decrypt those copies forever. For a friendly teardown (instance simply retired) that is acceptable. If the instance may have been compromised, follow up with =sops rotate -i= on each ephemeral-class file /and/ rotate the underlying secret values themselves.

* Recreated instances — the fork
Cloud instances and workspaces get a brand-new SSH host key every time they are recreated, which invalidates their enrolled recipient. Two paths, decided at provision time:

- *Seed the old host key* :: Provision with the remote-provisioning flow's =--seed-host-key= (see [[file:../../air/v0.1/remote-provisioning.org]]), injecting the previously-enrolled ed25519 host key via =--extra-files=. The instance keeps its age identity: no =.sops.yaml= change, no =updatekeys=, no private-repo commit. The cost is that private key material transits the provisioning machine — acceptable when the provisioner is a trusted personal machine and the window is short, but it is a real handling of a real private key.
- *Fresh key, revoke + enrol* :: Let the instance generate a new host key, then =secrets-revoke <name>= followed by =secrets-enroll <name> <target>=. Cleaner — no key material ever leaves the instance — at the cost of one =updatekeys= round and a registry row of churn. The registry deliberately keeps both rows, so recreation frequency stays visible.

Default to the fresh-key path; reach for seeding when re-provisioning is frequent enough that the sops churn becomes noise, and record that choice in the instance's own notes.

* Periodic sweep — registry vs reality
Stale recipients are both hygiene and security debt, so reconcile the registry against the actual fleet on a regular cadence (and opportunistically whenever an enrolment or revocation has you in the file anyway):

1. List active rows (empty =Revoked= column) in =ephemeral/registry.org= in the private repo.
2. For each, confirm the instance still exists and still needs its secrets.
3. Anything that fails the check: =nix run .#secrets-revoke -- <name>=.

Also eyeball the size of the enrolled block in =.sops.yaml= while there. It should stay small enough to read in one glance — if it grows past a handful of entries, that is the signal the fleet needs a different trust model, not more enrolments.

* Private-repo interface
The tooling in this repo is generic; everything identifying lives in the private repo, which must provide the following shape.

** Secret classes in =.sops.yaml=
Two creation rules, split by path:

- =core/**= :: YubiKey recipients only. Never touched by this tooling; never gains other recipients.
- =ephemeral/**= :: YubiKey recipients /plus/ the currently-enrolled instances. Holds only what keyless instances genuinely need (e.g. the atuin sync key, selected service tokens) — never the core set.

The enrolled recipients live between two sentinel comments inside the ephemeral rule — the only region the tooling ever writes to. Structural YAML editing (yq-style round-trippers) was rejected because =.sops.yaml= is hand-maintained with anchors and comments that round-trippers reformat; a fixed-format managed block edited line-by-line preserves the rest of the file byte-for-byte:

#+begin_src yaml
creation_rules:
  - path_regex: ^core/.*\.yaml$
    key_groups:
      - age:
          - *yubikey_a
  - path_regex: ^ephemeral/.*\.yaml$
    key_groups:
      - age:
          - *yubikey_a
          # BEGIN ENROLLED-INSTANCE RECIPIENTS (managed by secrets-enroll; do not edit by hand)
          - age1... # <name> (enrolled <date>)
          # END ENROLLED-INSTANCE RECIPIENTS
#+end_src

The sentinel pair must appear exactly once; the apps verify this and refuse to run otherwise.

** Registry
=ephemeral/registry.org=, created by the first enrolment, machine-maintained thereafter. One table, append-only:

| Name | Recipient | Created | Revoked | Purpose |

An active enrolment is a row with an empty =Revoked= column. Revocation fills the column rather than deleting the row, so the file plus its git history doubles as the enrolment audit log — including the churn of recreated instances.

** What the apps touch
Only =.sops.yaml= and =ephemeral/**=. Those paths must be clean in the checkout before running (the commit must contain exactly one enrolment or revocation); the rest of the checkout may be dirty. =updatekeys= runs over =ephemeral/**.{yaml,yml,json,env}= files that carry a sops metadata block, skipping anything else with a warning.
